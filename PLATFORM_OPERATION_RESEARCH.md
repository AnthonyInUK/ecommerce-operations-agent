# 平台内部电商运营分析研究

> 这份文档用于支撑电商分析 Agent 的业务定位。它不声称还原阿里、京东、拼多多的内部系统，而是基于公开资料，提炼平台电商运营分析中较稳定出现的分析对象、问题拆解方式和交付形态。

## 1. 研究边界

这份研究只做三件事：

1. 从公开商家工具、开放平台、帮助中心、产品页中提炼电商运营分析对象。
2. 将这些对象抽象成平台内部运营可能也会关注的工作流。
3. 把公开资料能支持的部分和项目能力做映射。

这份研究不做三件事：

1. 不假设拿到了三家公司内部真实 SOP。
2. 不复刻三家公司内部数据表、指标平台或权限体系。
3. 不把商家侧工具直接等同于平台内部运营系统。

更稳的判断是：

公开资料主要暴露的是商家/卖家侧工具，但这些工具反映了电商经营中共通的指标和维度。平台内部运营、类目运营、行业运营、增长运营也会围绕类似对象做更高粒度、更跨团队的分析。

## 3. 平台分析卡

### 3.1 阿里 / 淘宝天猫

公开资料强度：高。

| 项目 | 内容 |
|---|---|
| 核心指标 | 店铺核心数据、实时/昨日/近期数据、访客、浏览、收藏、加购、下单、支付、销售目标完成进度、物流进度、售后情况、行业排名。 |
| 核心维度 | 商品、品类、行业、搜索词、人群、渠道、终端、地域、页面、竞店、活动阶段。 |
| 常见下钻路径 | 店铺/活动大盘 -> 流量来源 -> 商品/单品表现 -> 收藏/加购/下单/支付 -> 地域/人群/页面/竞品。 |
| 典型异常场景 | 实时流量或交易波动、活动目标未达成、重点商品转化异常、库存预警、物流超时、售后异常、竞店排名变化。 |
| 输出形态 | 生意参谋看板、实时直播大屏、数据作战室、自助取数、专题工具、活动复盘。 |
| 我项目里对应的 Tool / Path | `GmvQueryTool`、`RegionPerformanceQueryTool`、`CategoryRankTool`、`FunnelAnalysisTool`、`RefundAnalysisTool`、`gmv_root_cause`、日报、异常巡检。 |

保守结论：

阿里公开工具体现出较完整的经营分析闭环：看大盘、看流量、看商品、看交易漏斗、看售后，再做活动复盘。它给项目的启发是：电商分析 Agent 不能只回答销售额，而要能串起交易、流量、商品、地域和售后。

| 项目 | 内容 |
|---|---|

| 核心维度 | 商品、新品、爆款、店铺、搜索流量、推荐流量、付费广告、内容营销、行业、类目、客服、物流。 |
| 常见下钻路径 | 店铺/商品经营 -> 流量来源 -> 转化/销售 -> 行业/类目对比 -> 履约/客服/售后。 |
| 典型异常场景 | 商品转化不达预期、活动效果不好、流量渠道异常、履约问题、客服压力、售后问题、行业趋势变化。 |
| 输出形态 | 京东商智、商家学习中心课程、数据工具、行业运营内容、营销/广告工具、客服/履约工具。 |
| 我项目里对应的 Tool / Path | `GmvQueryTool`、`OrderQueryTool`、`CategoryRankTool`、`FunnelAnalysisTool`、`UserMetricTool`、`RefundAnalysisTool`、`gmv_root_cause`、异常巡检。 |

保守结论：

京东公开资料体现出“商品运营 + 流量运营 + 数据分析 + 客服履约”的组合。它给项目的启发是：经营诊断不是单点查询，而是商品、流量、营销、履约、客服多环节协同。

## 4. 共同分析对象

从三家公开资料中，可以比较稳地提炼出这些共同对象：

| 类型 | 共同分析对象 |
|---|---|
| 交易结果 | GMV/成交额、订单量、支付、销售额、客单价 |
| 流量转化 | 访客、浏览、搜索、推荐、活动、收藏、加购、下单、支付 |
| 商品经营 | 商品、SKU、品类、爆款、新品、库存 |
| 用户/人群 | 人群画像、买家、会员/私域、用户运营 |
| 履约售后 | 发货、物流、客服、退款、退换货、售后评价 |
| 竞争/行业 | 行业趋势、竞品/竞店、市场行情、排名 |

这些对象在商家侧和平台内部侧都会出现。区别在于：

- 商家侧更关注单店铺、单品牌、单商品的经营表现。
- 平台内部侧更可能关注行业、类目、区域、商家群、活动场、用户群和治理动作。

## 5. 平台内部运营分析逻辑

公开资料无法直接还原平台内部系统，但可以合理抽象出一条平台内部运营分析链：

```text
发现异常 -> 拆解维度 -> 形成解释 -> 推动处理
```

### 5.1 发现异常

可能关注：

- GMV / 成交额下滑
- 订单量下滑
- 活动目标未达成
- 某区域、类目、商品表现异常
- 转化率下降
- 退款/售后升高
- 履约或客服压力异常

项目对应：

- `GmvQueryTool`
- 定时日报
- GMV 异常巡检
- 多基线异常判断

### 5.2 拆解维度

常见拆解方式：

- 交易结构：订单量、客单价
- 区域：地域/城市/区域
- 商品/品类：类目、SKU、爆款、新品
- 流量转化：访客、浏览、收藏、加购、下单、支付
- 用户/人群：活跃买家、人群画像、私域/会员
- 售后履约：退款、退换货、客服、物流

项目对应：

- `OrderQueryTool`
- `RegionPerformanceQueryTool`
- `CategoryRankTool`
- `UserMetricTool`
- `FunnelAnalysisTool`
- `RefundAnalysisTool`

### 5.3 形成解释

典型解释问题：

- 是单量问题还是客单价问题？
- 是流量不足还是转化承接差？
- 是某个品类/商品拖累，还是整体下滑？
- 是用户规模回落，还是活跃买家转化不足？
- 售后/退款是否成为主因？
- 当前证据是否足够，还是需要继续人工排查？

项目对应：

- `gmv_root_cause`
- `RootCauseAnalysisBuilder`
- `RootCauseAnalysisResult`
- `analysis_trace`
- `data_lineage`

### 5.4 推动处理

常见交付方式：

- 日报/周报
- 活动复盘
- 异常提醒
- 运营建议
- 通知类目、商家、活动、客服或履约团队

项目对应：

- 定时日报 trigger
- GMV 异常巡检 trigger
- `send_notification`
- 飞书 webhook / IDE log 降级
- 防重复推送、持久化幂等、只读保护、缓存、超时、降级

## 6. 角色视角

| 角色 | 关注指标 | 关注维度 | 异常场景 | 常见动作 | 项目对应 |
|---|---|---|---|---|---|
| 平台运营 | GMV、订单、活动目标 | 区域、类目、活动、商家群 | 大盘或活动下滑 | 拉相关团队排查、发日报/周报 | `GmvQueryTool`、日报、异常巡检 |
| 类目/行业运营 | 类目 GMV、商品转化、库存、价格带 | 品类、商品、SKU、商家、价格带 | 某类目或商品下滑 | 调活动资源、推商家优化 | `CategoryRankTool`、`gmv_root_cause` |
| 增长/营销运营 | 流量、转化、ROI、活跃买家 | 渠道、人群、活动、内容场 | 渠道质量差、转化下降 | 调投放、人群、素材和活动 | `UserMetricTool`、`FunnelAnalysisTool` |
| 售后/治理运营 | 退款率、退换货、客服压力、物流异常 | 商品、商家、类目、区域 | 退款升高、物流/客服异常 | 介入治理、通知商家或客服 | `RefundAnalysisTool`、异常通知 |
| 数据分析师 | 指标口径、归因、报告 | 多维度组合 | 复杂归因需求 | 出报告、沉淀口径和案例 | `verified cases`、`analysis_trace`、`RootCauseAnalysisResult` |

## 7. 对项目的设计约束

这份研究对项目有三个约束：

1. 不追求复刻任何一家内部数据库。
2. 要贴合电商运营分析对象：GMV、订单、用户、品类、区域、漏斗、退款、履约。
3. 要贴合分析工作流，而不是只做单轮问数。

因此项目主线应该继续围绕：

```text
发现异常 -> 拆解维度 -> 形成解释 -> 推动处理
```

当前项目已经覆盖：

- 发现异常：日报、GMV 异常巡检、多基线判断。
- 拆解维度：区域、订单、用户、品类、漏斗、退款。
- 形成解释：`gmv_root_cause`、结构化 root cause DTO、analysis trace。
- 推动处理：通知、降级、幂等、去重、只读、缓存。

目前仍需补强：

- 更真实的履约/物流口径。
- 更细的售后原因分类。
- 更清晰的责任对象和处理建议。
- 对证据不足的显式表达。
- 平台内部岗位 JD 的进一步佐证。

## 8. 后续研究建议

下一步如果继续补证据，优先收集这几类资料：

1. 平台运营、类目运营、行业运营、增长运营、数据分析师岗位 JD。
2. 财报、投资者材料、行业报告中对 GMV、用户、商家、履约、广告、退货的描述。
3. 平台规则中关于发货、退款、退换货、商家服务分、商品质量的条款。
4. 公开技术分享中关于经营诊断、智能客服、推荐搜索、广告投放、供应链履约的系统实践。

这些资料可以继续帮助项目从“商家侧公开工具抽象”往“平台内部运营分析逻辑”靠近。

财报材料能支持的保守结论：

平台内部运营分析大概率不只看 GMV，而会同时看用户规模、购物频次、品类结构、广告/营销效率、履约效率、商家生态和售后治理。因此项目里的 `UserMetricTool`、`CategoryRankTool`、`FunnelAnalysisTool`、`RefundAnalysisTool`、日报和异常巡检都不是凭空设计出来的能力。

## 13. 研究结论更新

结合商家工具、岗位 JD、财报/投资者材料、平台规则和技术分享，可以把项目定位进一步收紧为：

> 这个项目不是复刻某家平台内部系统，而是基于公开资料抽象电商平台运营分析中稳定存在的工作流：发现异常、拆解维度、形成解释、推动处理，并尝试用 Text-to-Code 把这条链变成可执行、可追踪、可复用的 Agent 工作流。

当前证据最能支撑的项目能力是：

| 工作流阶段 | 公开证据支撑 | 项目已有能力 |
|---|---|---|
| 发现异常 | 财报中的 GMV/用户/品类/广告/履约指标；岗位中的日报、周报、预警职责 | 高频问数、定时日报、GMV 异常巡检 |
| 拆解维度 | 商家工具中的商品、品类、流量、地域、漏斗、售后；岗位中的投放、竞品、活动复盘 | 区域、订单、用户、品类、漏斗、退款 Tool |
| 形成解释 | 京东经营诊断 Multi-Agent 分享；岗位中的数据拆解、复盘和策略优化 | `gmv_root_cause`、`RootCauseAnalysisBuilder`、`analysis_trace` |
| 推动处理 | 平台规则和客服/售后/履约材料；岗位中的跨团队协作和方案落地 | trigger、通知、降级、幂等、去重、只读和缓存 |

目前仍然最需要补强的地方：

1. 履约/物流异常：目前项目还没有独立 Tool。
2. 广告/投放 ROI：目前只在用户/漏斗层有雏形。
3. 商家/商品治理：目前只通过退款和品类侧间接体现。
4. 责任对象分发：异常后还没有明确到类目运营、商家运营、售后治理等角色。
5. 更强证据口径：公开 Olist 数据缺少真实行为流和售后明细，需要继续用逻辑真实的 demo 口径补齐。
