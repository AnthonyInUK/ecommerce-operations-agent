# Daily Report Demo

## 这条演示链在讲什么

这不是“用户问一句，Agent 答一句”的链路，而是：

1. 到达设定时间
2. 系统自动触发日报任务
3. 自动汇总昨日核心经营指标
4. 自动整理成可读的日报格式
5. 通过 `send_notification` 输出到飞书或默认通知通道

业务上，它解决的是：

- 运营每天都要看昨日经营概览
- 但不应该每天都靠人工重新发起查询
- 日报应该稳定、统一、可追溯

## 当前日报包含什么

### 一、核心看板

- GMV
- 支付订单量
- 客单价
- DAU
- 活跃买家

### 二、结构观察

- Top 区域
- Top 品类

### 三、系统备注

- 当前日报只做“昨日核心经营概览”
- 异常检测和 root cause 由另一条巡检 trigger 负责
- 会带上数据源说明，方便区分 demo 与 Olist 支线

## 演示时怎么讲

可以直接这样讲：

> 前面几周我做的是被动分析链，用户问什么，系统答什么。  
> 到 Week E，我先从一个最稳定、最高频、最容易验证的主动动作开始，也就是“定时日报”。  
> 它不是让模型自由发挥，而是到点自动拉取昨日核心经营指标，再整理成固定日报格式，通过通知通道发出去。  
> 这样系统第一次从“会分析”变成“会按时交付分析结果”。

## 当前边界

- 这条日报链目前以“昨日经营概览”为主
- 不直接承担异常归因任务
- 不承担复杂多轮解释
- 更像稳定的运营看板推送

## 和异常巡检的分工

- `daily_report`：稳定输出昨日经营概览
- `gmv_drop_watch`：发现异常后自动触发 root cause

所以这两条 trigger 在产品上是分工关系，不是重复建设。
