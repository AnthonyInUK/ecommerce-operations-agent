# 博客写作手册：给未来的 Claude / Codex / 自己

## 这份文件是干什么的

这不是项目文档，也不是系列博客目录。  
这份文件的作用是：

- 帮我判断一个主题现在能不能写成博客
- 帮我把博客写成“技术博客”，而不是“开发流水账”
- 给未来的 Claude / Codex 一个固定参考模板
- 让后续新博客的质量和风格尽量一致

如果以后要继续写这个项目的系列博客，优先先看这份文件，再动手。

---

## 核心原则

### 1. 先写“为什么”，再写“怎么做”
每篇博客都要先回答：

- 这个问题为什么值得解决
- 现有方案为什么不够
- 我为什么做出现在这条技术路线选择

不要一上来就讲类名、模块名和代码结构。

### 2. 先写“能力边界”，再写“功能清单”
博客不是 changelog。  
不要写成：

- 我加了什么 Tool
- 我改了什么 YAML
- 我补了什么 validator

更好的写法是：

- 这个能力解决什么问题
- 它和现有路线相比，边界在哪里
- 这个设计为什么比另一种设计更合适

### 3. 一篇文章只证明一个核心判断
每篇文章只能有一个主论点。

例如：
- 为什么电商运营分析更适合 Text-to-Code
- 为什么业务型 Agent 的 Tool 不该按表切
- 为什么“为什么跌了”必须收成标准深路径

不要一篇里同时塞五个论点。

### 4. 代码只拿来“证明”，不要拿来“堆密度”
代码片段的作用是：

- 证明这个设计真的存在
- 让读者看到它大概长什么样

不要把博客写成源码 walkthrough。  
一篇文章里通常只放 1 到 3 个短片段。

### 5. 读者默认懂一点 LLM，但不懂这个项目
默认读者：

- 对 LLM / Agent 感兴趣
- 可能知道 LangChain / Coze / 工作流
- 但没接触过 AssistantAgent
- 也没跟过当前项目开发过程

所以第一次出现的术语要轻解释：

- `Text-to-Code`
- `Experience`
- `FastIntent`
- `root cause`
- `raw -> dwd -> dim -> ads`

---

## 什么时候一个主题“够资格”单独成文

一个主题至少满足下面 3 条，再单独写博客：

1. 已经有**真实实现**
2. 已经有**明确设计取舍**
3. 已经有**至少一个真实坑或边界**

如果只有功能，没有取舍和坑，这个主题通常还不够厚。

## 现在这个项目里，什么主题适合单独写

### 已成熟可写

1. 从问答 Demo 到最小业务闭环
2. 为什么电商运营分析更适合 Text-to-Code，而不是纯 NL2SQL
3. 怎么把“为什么跌了”做成一条标准深路径
4. 从被动问答到主动日报和异常巡检

### 后续可扩展

1. Experience / FastIntent
2. 多轮理解 / 最小业务记忆
3. Olist 数据迁移
4. 评测闭环 / bad case 回流

### 暂不建议单独成文

1. 只讲某个单独 Tool 的实现
2. 只讲某个 YAML 文件
3. 只讲某个 validator
4. 只讲单个类的源码分析

这些内容更适合成为某篇文章里的一个小段，而不是整篇主题。

## 每篇博客的推荐结构

不是每篇都必须完全一致，但优先用下面这套骨架。

### 1. 背景 / 问题定义
回答：

- 这个问题为什么存在
- 谁在受这个问题影响
- 为什么值得解决

### 2. 现有方案 / 路线分析
回答：

- 公开可见的现有路线有哪些
- 它们各自更适合解决什么
- 边界是什么

### 3. 我的路线选择
回答：

- 我为什么没有选 A，而是选了 B
- 这个选择背后的关键判断是什么

### 4. 具体实现
回答：

- 这个能力最后落成了什么结构
- 系统链路怎么跑
- 必要时给一个很短的代码或配置片段

### 5. 踩坑与修正
回答：

- 最重要的 2 到 4 个坑是什么
- 我怎么修的
- 这件事如何改变了设计

### 6. 总结
回答：

- 这个主题最后验证了什么
- 对业务型 Agent 的认知有什么变化

## 标题怎么起

### 系列格式
统一使用：

`AssistantAgent 电商分析 Agent 系列 0X：副标题`

### 好标题的标准

- 有主题，不空
- 有论点，不是功能名
- 不要太学术
- 不要太“项目周报”

### 标题例子

- AssistantAgent 电商分析 Agent 系列 01：从问答 Demo 到最小业务闭环
- AssistantAgent 电商分析 Agent 系列 02：为什么电商运营分析更适合 Text-to-Code，而不是纯 NL2SQL
- AssistantAgent 电商分析 Agent 系列 03：怎么把“为什么跌了”做成一条标准深路径
- AssistantAgent 电商分析 Agent 系列 04：从被动问答到主动日报和异常巡检

---

## 语言风格要求

### 要的风格

- 克制
- 具体
- 有判断
- 有边界感
- 有真实工程感

### 不要的风格

- 空泛热血
- 自夸
- 过度断言别人“不行”
- 机械流水账
- AI 腔太重

### 少用的表达

- 我后来意识到……
- 我越来越明确……
- 这让我很受启发……
- 这个非常炸裂……

不是完全不能用，而是不要全篇都是这种句子。

### 更好的表达

- 更准确的说法是……
- 从公开形态看……
- 这个选择背后的判断是……
- 这里真正的边界在于……
- 我最后做的收敛是……

---

## 引用和证据怎么放

### 可以引用什么

- 公开产品资料
- 官方文档
- 官方仓库
- 第三方测评
- 本项目代码 / 配置 / 演示链

### 不要写成什么

- “大厂做不到”
- “他们内部一定是这样”
- “一线员工都这么说”

如果没有一手证据，就写成：

- 从公开可见的产品形态看……
- 公开资料里更容易看到……
- 这更像一条公开可见的技术路线……

---

## 代码片段使用规则

### 适合放代码的场景

1. 标准深路径结构
2. 主动触发保护层配置
3. 一段非常短的 Tool 调用链

### 不适合放代码的场景

1. 大段类实现
2. 大段 YAML
3. 一个文件从头贴到尾

### 代码长度建议

- 单段尽量 5 到 12 行
- 只证明设计，不解释全部实现

## 写作流程

以后每写一篇博客，都按这个流程：

1. 先在 `BLOG_SERIES_PLAN.md` 里确认主题是否足够成熟
2. 先写一句主论点
3. 用 6 节结构搭骨架
4. 先写“背景”和“路线选择”
5. 再写实现
6. 最后补“踩坑”
7. 检查有没有写成项目周报
8. 检查术语第一次出现有没有解释
9. 检查是否需要加 1 到 2 个代码片段
10. 最后补一句：这是阶段性总结还是完结稿

## 给 Claude / Codex 的固定提示模板

以后如果要让我或别的模型继续写这个系列，优先把下面这段当作提示词使用：

```md
你现在在帮我写 AssistantAgent 电商分析 Agent 系列博客。

写作目标：
- 写成技术博客，不要写成开发流水账
- 重点讲问题定义、路线选择、能力边界、设计取舍和真实踩坑
- 默认读者懂一点 LLM/Agent，但不懂这个项目，也没接触过 AssistantAgent

写作要求：
- 一篇文章只证明一个核心判断
- 先写为什么，再写怎么做
- 第一次出现术语时做轻解释
- 代码只作为短证据，不做源码 walkthrough
- 不要替大厂下过强结论，只能基于公开形态做判断
- 少用“我后来意识到 / 我越来越明确 / 这让我很受启发”这类 AI 腔表达
- 风格要克制、具体、有真实工程感

文章结构优先使用：
1. 背景 / 问题定义
2. 现有方案 / 路线分析
3. 我的路线选择
4. 具体实现
5. 踩坑与修正
6. 总结

如果主题还不够成熟，不要硬写成完整文章，要明确指出当前更适合作为哪篇文章中的一节。
```
