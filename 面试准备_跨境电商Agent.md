## 0. 30 秒电梯陈述（开场背这个）

> "我做的是一个**电商运营诊断 Agent**，不是聊天机器人，而是一条端到端工作流：**发现异常 → 多维归因 → 责任分发 → 生成通知草稿**。
> 技术上走的是 **Code-as-Action** 路线——模型自己写 Python、在 **GraalVM 沙箱**里执行，灵活组合多个工具，而不是只做 Text-to-SQL。
> 我设计了**三条路径**：高频问题走规则快路径（不烧大模型 key），复杂归因走大模型深路径，再加一条 **trigger 主动巡检**（不等人问，自动发现 GMV 异常）。
> 因为 Agent 真正的瓶颈是 **LLM provider 调用**，我手写了一层**零依赖韧性层**（限流→并发隔板→熔断→重试→超时），压测能扛住 provider 80% 故障而不雪崩。
> 整套架构是平台无关的——换成跨境的选品、Listing 优化、售后，工具换一批、数仓接口换一下就能复用。"

记住三个关键词锚点：**Code-as-Action / 三路径路由 / 韧性层**。这三个是你和其他实习生拉开差距的地方。

## 0.1 最新版整体业务逻辑（先按这条讲）

这版项目不要先讲成“我做了一个能聊天的 Agent”，而要讲成：

> 我做的是一个面向电商运营异常处理的业务型 Agent。它的主线不是“用户问一句，模型答一句”，而是把真实运营里的异常处理流程串起来：**发现异常 -> Agent 归因 -> 人工确认 -> 责任派发 -> 飞书通知 / 处理闭环**。

最完整的业务链路是：

```text
BI / 监控规则发现异常
-> 生成 anomaly_id
-> Java 根据 anomaly_id 找到异常对象
-> Java 根据 metric_id 判断异常类型
-> Java 选择对应 analysis_route
-> Java 执行只读 Tool，拿到结构化证据
-> LLM 基于证据做解释、主因排序、责任分发和通知建议
-> Java 校验输出格式和安全边界
-> 数据分析师人工确认
-> 派发给平台运营 / 类目运营 / 增长运营 / 售后治理
-> 飞书通知或降级到日志
-> 业务负责人接手、补充处理记录、关闭异常或标记误报
```

这里的核心不是让 LLM 全权接管业务系统，而是把它放在一个受控位置里：

```text
Java 后端：负责路由、查数、校验、安全边界和状态流转
Tool 层：负责只读查询，返回结构化证据
LLM：负责计划建议、解释证据、主因排序、责任分发和通知草稿
人工：负责确认、派发、接手、关闭和误报判断
```

### anomaly_id 是什么

`anomaly_id` 可以理解成业务系统里一条异常的唯一编号。比如：

```text
anom-20180829-east-gmv
```

它代表：

```text
2018-08-29 华东区域 GMV 出现异常
```

真实业务里，这个 ID 可能来自 BI 监控、指标巡检、告警平台或运营规则。Agent 不一定负责“发现所有异常”，它更像接住这条异常之后继续往下分析：

```text
异常已经出现了
-> Agent 判断是什么类型
-> Agent 查证据
-> Agent 给出处理建议
-> 人决定要不要派发和通知
```

这样设计更贴近真实企业系统，因为中大厂通常已经有 BI、数仓和监控系统，Agent 更适合接在这些系统后面，补上“自动归因和推动处理”这一层。

### Java 怎么“理解”异常

Java 不是像人一样理解业务，而是做规则匹配和字段映射。

比如当前异常对象是：

```json
{
  "anomaly_id": "anom-20180829-east-gmv",
  "metric_id": "gmv",
  "date": "2018-08-29",
  "region": "华东",
  "current_value": 1870.87,
  "previous_value": 3841.30,
  "change_rate": -51.3
}
```

Java 会根据 `metric_id = gmv` 找到 GMV 异常对应的分析路线：

```text
GMV 异常
-> 区域表现
-> 订单结构
-> 用户规模
-> 品类拖累
-> 漏斗转化
-> 退款压力
```

所以 Java 的“理解”本质是：

```text
根据 anomaly_id 找异常对象
根据 metric_id 找业务规则
根据异常类型选择 route
根据 route 选择允许调用的 Tool
根据 Tool 结果整理证据
```

### LLM 具体做什么

LLM 不直接查数据库，也不直接改业务状态。它主要做四件事：

1. **计划建议**：在 Java 给出的异常类型、业务规则和可用 Tool 范围内，生成候选分析步骤。
2. **解释证据**：把 Tool 返回的数字解释成业务语言。
3. **主因排序**：判断哪些线索更像主因，哪些只是辅助信号。
4. **生成通知草稿**：把结论整理成业务负责人能看懂的通知内容。

举例，Tool 查出来：

```text
GMV：3841.30 -> 1870.87
订单量：40 -> 13
客单价：96.03 -> 143.91
DAU：10 -> 8
家居品类贡献：1711.96 -> 0
退款金额：60.00
漏斗转化：0.50 -> 0.50
```

LLM 要把这些证据解释成：

```text
GMV 下滑主要不是客单价导致，因为客单价反而上升。
更明显的信号是订单量下降，同时用户规模也回落。
品类侧家居贡献下降最大，因此应优先排查家居品类下的商品、商家、库存和活动资源。
漏斗转化基本稳定，所以暂时不作为主因。
退款有压力，但不是第一主因。
```

### Java 为什么还要做边界检查

LLM 输出的 plan 或结论不能直接被信任。Java 会检查：

```text
JSON 格式是否正确
Tool 是否在白名单里
是否出现写操作或危险操作
步骤数是否超过限制
是否符合当前异常类型
通知建议是否在允许枚举里
输出字段是否完整
```

如果 LLM 输出不合规，比如：

```json
{
  "tool": "DeleteOrderTool",
  "reason": "删除异常订单"
}
```

Java 会直接拒绝，因为这个 Tool 不在只读白名单里。然后系统会 fallback 到固定 root cause 链路。

这套设计的核心价值是：

```text
LLM 负责灵活性和表达
Java 负责稳定性、安全性和可追责
```

### Fast Path 和 Deep Path 怎么分

项目里不是所有问题都交给 LLM 自由推理，而是分成两类。

**Fast Path** 处理高频、标准、低歧义问题：

```text
昨天 GMV 多少？
订单量 / 客单价怎么样？
DAU 和活跃买家多少？
华东 GMV 多少？
```

这类问题口径稳定，直接走：

```text
用户问题
-> FastIntent 匹配意图
-> 解析时间 / 指标 / 维度
-> 调用对应 Tool
-> 固定结构回答
```

它本质上是：

```text
intent -> tool -> params -> answer template
```

这不是“低级写死”，而是把高频稳定业务动作产品化。业务系统里标准问数要快、稳、便宜，不应该每次都让 LLM 从零规划。

**Deep Path** 处理复杂归因问题：

```text
华东 GMV 为什么跌了？
为什么订单量下降？
这个异常要不要发飞书？
哪些负责人优先处理？
```

这类问题需要多步下钻：

```text
用户问题 / anomaly_id
-> 判断是 root cause / 复杂分析
-> 选择 deep route
-> 顺序执行多个 Tool
-> 汇总结构化证据
-> LLM 解释证据、排序主因、生成责任分发
-> 人工确认后进入工作流
```

GMV 下滑的 deep path 是：

```text
RegionPerformanceQueryTool
-> OrderQueryTool
-> UserMetricTool
-> CategoryRankTool
-> FunnelAnalysisTool
-> RefundAnalysisTool
```

面试里可以这样说：

> Fast Path 把确定性高频问题固定下来，Deep Path 负责复杂归因。业务型 Agent 不是越自由越好，而是该稳定的地方稳定，该智能的地方智能。

### GMV root cause 怎么判断主因

GMV 最底层先按公式拆：

```text
GMV = 支付订单量 × 客单价
```

所以第一层先判断：

```text
是订单量下降？
还是客单价下降？
```

如果订单量下降，系统继续往下查：

```text
用户规模是否下降
转化漏斗是否下降
品类 / 商品是否拖累
活动 / 投放 / 流量是否变弱
```

如果客单价下降，系统继续查：

```text
高价商品是否减少
低价品类占比是否上升
优惠力度是否变大
价格是否变化
```

如果退款压力上升，系统继续查：

```text
退款集中在哪个品类
退款集中在哪些商品 / 商家
是否物流延迟
是否商品质量问题
是否客服处理异常
```

在当前样例里，主因排序可以这样理解：

```text
1. 订单量下降：GMV 直接组成项，影响最大
2. 家居品类拖累：家居从前一日贡献明显变成当前几乎无贡献
3. 用户规模下降：DAU 和活跃买家同步下降，说明流量或人群触达可能变弱
4. 退款压力：有售后压力，但不是第一主因
5. 漏斗转化：转化率基本稳定，所以不作为主因
```

更通俗地说：

```text
先看 GMV 公式里的直接因素
再看哪个业务维度贡献下降最大
再看用户、漏斗、退款这些辅助证据
最后把原因按处理优先级排出来
```

### 如果是其他异常，系统怎么查

不同 `metric_id` 会对应不同分析重点。

```text
GMV 异常：
完整 root cause，查区域、订单、用户、品类、漏斗、退款

订单量异常：
优先查订单量、支付订单数、转化、流量，不把退款放第一位

品类 GMV 异常：
优先查品类、商品、商家、库存、活动资源

退款率异常：
优先查退款金额、退款订单、退款原因、物流和客服

转化率异常：
优先查浏览、加购、下单、支付漏斗
```

这个设计的好处是：

```text
不是所有异常都跑同一套 GMV root cause
而是 anomaly_id -> metric_id -> analysis_route
不同异常走不同排查重点
```

### 责任分发的业务逻辑

系统不会只输出“异常了”，而是要回答：

```text
谁应该先看？
为什么是他？
他要查什么？
下一步产出是什么？
```

比如 GMV 下滑时：

```text
平台运营：
确认异常影响范围，是只有华东掉，还是大盘也掉。

经营分析 / 平台运营：
拆 GMV = 支付订单量 × 客单价，确认主要拖累来自单量还是客单价。

类目运营：
定位家居品类下的重点商品、商家、库存和活动资源。

增长运营：
检查渠道流量、活动曝光、人群触达和投放节奏。

售后治理：
排查退款、物流履约、客服处理和退换货原因。
```

优先级可以这样理解：

```text
P0：必须先处理的主责角色
P1：需要协同补证据的角色
P2：暂时观察，不一定立刻介入
```

### 工作流状态怎么走

这条线不是“飞书发出去就结束”，而是有处理闭环：

```text
待确认
-> 已确认
-> 已派发
-> 处理中
-> 已关闭
```

同时通知状态单独存在：

```text
未发送
-> 已发送飞书
-> 发送失败
-> 去重拦截
-> 降级日志
```

这两个状态必须分开：

```text
通知成功，不代表业务已经处理
业务已派发，也不代表飞书一定发送成功
```

所以你的页面里应该让面试官看到：

```text
数据分析师视角：确认异常、分析 root cause、派发责任人
业务负责人视角：看到分给自己的异常、接手处理、补充记录、关闭异常
管理视角：看所有异常的状态、主责分布、处理进度和误报情况
```

### 这套设计为什么虽然麻烦但合理

如果只是 Demo，可以让 LLM 直接回答：

```text
为什么 GMV 跌了？
```

但真实业务里会遇到：

```text
LLM 选错 Tool 怎么办？
LLM 输出格式漂移怎么办？
LLM 把退款误判成主因怎么办？
LLM 误报后直接发飞书怎么办？
业务方追问“依据是什么”怎么办？
服务重启后状态丢了怎么办？
```

所以项目里的复杂度来自真实业务约束：

```text
稳定性
安全性
可解释性
可评测
可追责
可接入现有业务系统
```

面试总结句：

> 这个项目里我没有把 LLM 当成一个万能执行器，而是把它放在受控业务工作流里。标准问题走 Fast Path，复杂异常走 Deep Path；Java 负责边界和状态，Tool 负责查数，LLM 负责证据解释、主因排序和通知草稿，人工负责最终确认。这样牺牲了一部分自由度，但换来了业务系统需要的稳定性、安全性和可追责。

## 2. Code-as-Action + GraalVM 沙箱

**是什么**：复杂问题不走预定义工具的固定编排，而是**让大模型直接生成 Python 代码**，在沙箱里执行；代码里可以自由调用、组合多个工具（如先查 GMV 再拆订单再算客单价）。入口 `POST /api/ecommerce/agent-run`。

**为什么不只做 Text-to-SQL**：
- 运营归因是**多步、有分支**的（查 GMV→发现跌→拆是订单量还是客单价→再下钻区域/品类→看用户规模/漏斗/退款）。SQL 模板写死扛不住组合爆炸。
- Code-as-Action 让模型用"写代码"的方式编排工具，灵活度高一个量级。这是 CodeAct / OpenInterpreter 那一派的思路。

**追问1：沙箱怎么保证安全？**（⚠️ 这里别吹过头，源码我看过）
> "沙箱是框架的 GraalVM polyglot context。隔离靠三点：**`allowIO=false`（不能碰文件/网络）、`allowNativeAccess=false`（不能调本地库）、30 秒执行超时**。host access 这块默认是开的，所以真正的安全边界其实是**能力白名单**——沙箱里只能看到我**桥接进去的工具对象**，看不到任意 classpath。如果要上生产、跑真正不可信的代码，我会进一步收紧 host access 白名单 + 加资源配额（GraalVM 的 sandbox resource limits）。"
> （这样答：既懂原理，又知道它的局限和下一步——比硬吹"绝对安全"强得多。）

**追问2：模型生成的代码出错/跑飞怎么办？**
> "超时兜底 + 整个 LLM 调用包在韧性层里；执行记录（ExecutionRecord）会留痕，配合 SQL 审计能追到每个数字是哪条 SQL 算出来的，把 Text-to-Code 从黑盒变成可审计。"

**诚实边界**：GraalVM 执行器是上游的，我没改它的内核，我做的是上层工具桥接和调用编排。

## 3. 三条路径路由（这是你架构设计能力的体现，重点讲）

| 路径 | 触发 | 怎么走 | 为什么 |
|---|---|---|---|
| **Fast（快路径）** | 高频标准问题（"昨天 GMV 多少"） | FastIntent 规则路由 → 直接调单个工具 + ads 快照 | 低延迟、低成本、可复现、**不需要大模型 key** |
| **Deep（深路径）** | 开放/复杂归因 | `agent-run` → 模型写 Python → 沙箱执行多工具 | 灵活处理组合问题 |
| **Trigger（主动巡检）** | 定时/事件 | 自动跑 GMV 异常检测 → 命中则跑 root cause | Agent 的"自主性"，不等人问 |

**为什么分路径**（金句）：
> "不是所有问题都值得烧大模型。**80% 是高频标准问数**，走规则快路径又快又省又稳；只有**开放式归因**才值得动用大模型写代码。这是成本和能力的工程权衡——尤其对初创，token 成本是真金白银。"

代码佐证：`EcommerceQuestionAnswerService` 里按 `intent` switch，命中标准意图直接 `baseAnswer(..., "fast", ...)`，复杂的才进 `"deep"`。

**对这家公司的话术**：跨境同理——"查某 ASIN 昨天销量"走快路径，"为什么这个 listing 转化掉了"走深路径归因，"每天自动巡检广告 ACOS 异常"走 trigger。

## 4. 12 个诊断工具（强调：实时算、不写死）

## 5. Root Cause 归因链路

**是什么**：`RootCauseWorkflowTool` + `RootCauseAnalysisBuilder` 把多个工具的结果串成一份结构化结论：**关键证据 → 主要原因排序 → 责任分发 → 飞书通知草稿**。Demo 固定到一条可复现的 Olist 异常样本（`demo-report-date=2018-08-29`）。

## 6. ⭐ 韧性层（你最硬的工程亮点，准备最细）

**核心认知（先抛这句，立刻显出你想清楚了）**：
> "Agent 的瓶颈和故障点不在本地 CPU，而在 **LLM provider 调用**——超时、429、5xx、成本飙升。所以我说的'高并发'是指**扛住 provider 故障**，不是裸 QPS。"

**编排顺序（由外到内，每层都有动机）**：
```
限流 → 并发隔板 → 熔断 → 重试(退避+抖动) → 单次超时 → LLM调用
(成本/配额) (保线程)  (快速失败)  (救瞬时错误)    (给每次设预算)
```
- **限流（令牌桶）**：最先做准入，挡掉最便宜，护配额和成本。
- **并发隔板（Semaphore）**：限在途调用数，防慢调用堆积耗尽线程/内存。
- **熔断（CLOSED/OPEN/HALF_OPEN）**：provider 不健康时**连试都不试**，直接快速失败。计数滑动窗口（环形缓冲）算失败率，超阈值跳 OPEN；隔一段时间进 HALF_OPEN 放少量探活流量，好了再 CLOSED。
- **重试（指数退避 + 抖动）**：只对可重试的瞬时错误重试；抖动打散并发重试时间点，避免同步重试风暴。
- **超时**：最内层，给每次尝试套时间预算。

**最关键的设计细节（这句话能让面试官点头）**：
> "**重试放在熔断的内层**——也就是'一次逻辑调用（含若干次重试）= 熔断器的一个样本'。这样 provider 一旦挂，少量逻辑调用很快把熔断打开，之后**连重试都不再发生**，彻底止血，而不是无脑重试把故障放大成重试风暴。"

**压测数字（背下来，可复现脚本 `LlmResilienceLoadTest`，用假模型不烧钱）**：
- 健康态：**~3,900 QPS，p50 3.8ms / p95 5.0ms / p99 8.9ms** —— 包装开销很薄。
- 降级态（provider ~80% 失败）：熔断在 **~35 次调用后打开**，把剩余 **~2,960 个请求瞬间短路到降级文案**，只有 **~1%** 还在打挂掉的 provider；**零异常泄漏**。

**为什么自己手写不用 Resilience4j**（必被问）：
> "一是**零依赖**，框架已经很重，不想再引一个；二是**为了真正理解每层的交互**——尤其'重试在熔断内层'这个顺序，用现成库很容易配错。手写让我能把语义讲清楚、测试覆盖到位。生产上如果团队更熟 Resilience4j，我也能平移。"

**诚实边界**：限流是单机令牌桶，多实例要配合网关或分布式限流——这点我会主动提，显示我知道边界。

## 7. Prompt 注入防护

**是什么**：`PromptInjectionGuard`——在问题进入处理前做风险评估，命中高危模式（"ignore previous"、"system prompt"、"删表"、"打印系统提示词"、"绕过规则"等中英黑名单）就 **blocked**，记审计、返回拒绝，不进下游。

**怎么讲（别吹成完整方案）**：
> "这是**第一道轻量关卡**——关键词 + 规则，挡掉最明显的注入和越权（比如诱导删库、套系统提示词）。我清楚它不是完整方案：真正的防御要分层——**输入侧检测 + 沙箱能力白名单（模型就算被注入也只能调白名单工具）+ SQL 审计兜底**。我现在三层都有雏形，关键词只是最外层。"

这样答把"一个关键词列表"提升成"纵深防御的认知"。

## 12. 高频刁难问题 速答

**Q：这不就是个套壳吗？**
A：见第 1 节。主动划边界 + 强调难的部分（韧性/归因/测试）是我做的。

**Q：为什么用 Java 不用 Python？业界 Agent 不都 Python？**
A："框架是 Spring AI 生态所以 Java。我清楚 Agent 主流在 Python（LangGraph/LlamaIndex），我也熟。选 Java 让我把**工程化、并发、可观测**这些后端硬功夫做扎实了——韧性层这种东西在 Java 里更自然。**你们用什么栈我都能上手**，全栈我前后端都能独立做。"
（这家初创 JD 要全栈、要能独立落地，这么答正中下怀。）

**Q：你压测 3900 QPS，但真实 LLM 调用很慢，这数字有意义吗？**
A："好问题。3900 QPS 是**韧性层包装本身的开销**——我用假模型剥离了 provider 延迟，专门量'这层会不会成为瓶颈'，结论是 p99 不到 9ms，几乎无开销。真实场景 QPS 由 provider 决定，韧性层的价值在**降级态**：provider 挂了能瞬间短路兜底，那张图才是重点。"

**Q：trigger 主动巡检会不会误报刷屏？**
A：阈值 + verified/runtime bad cases 评测闭环防退化；命中才跑 root cause；生成的是**草稿**不是直接群发，留人工确认。（看 SECURITY.md / ROBUSTNESS.md）

**Q：如果让你从零给我们做选品 agent，第一周做什么？**
A："先打通最小闭环：接一个平台 API（如 SP-API）做数据层 → 定义 3-4 个选品工具（市场热度/竞品/利润/合规）→ 用 Code-as-Action 让模型编排出一份选品报告 → 加韧性层和审计。先做**一个站点一个品类的可复现 demo**，再横向扩。"
