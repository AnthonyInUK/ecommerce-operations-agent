# Ecommerce Operations Anomaly Agent Demo Script

This script is for a 3-5 minute interview demo. The goal is to explain the project as a small business workflow, not just a chat UI.

## 1. Opening

Open:

```text
http://localhost:18080/agent-console/index.html
```

One-sentence positioning:

> This is an ecommerce operations anomaly analysis Agent. It starts from a business anomaly ID, runs route-specific root cause analysis, then supports human confirmation, owner dispatch and Feishu notification.

## 2. Show The Main Chain

Point to the top story:

```text
发现异常 -> 拆解维度 -> 形成解释 -> 推动处理
```

Explain it in business language:

- `发现异常`: monitoring rules or BI jobs create an anomaly record.
- `拆解维度`: the Agent chooses the analysis route from `anomaly_id -> metric_id -> analysis_route`.
- `形成解释`: tools produce evidence, cause ranking and action routing.
- `推动处理`: humans confirm, dispatch owners, send Feishu and record workflow status.

## 3. Run The Showcase Anomaly

Click `运行 GMV 异常巡检`.

Expected sample:

```text
2018-08-29 华东 GMV 从 3841.30 下降到 1870.87
```

Explain:

> The anomaly list is not generated by a chat prompt. It represents what a real BI or monitoring system would push into an Agent layer. The Agent takes the anomaly ID and decides what to analyze.

## 4. Show Agent Capability

In `Agent 接入链路`, explain:

- `异常接入`: the upstream business system provides `anomaly_id`.
- `分析路由`: the Agent maps the anomaly to the right route.
- `Tool 取数`: the Agent gathers evidence through business tools.
- `业务交付`: the output becomes a notification draft and workflow record.

Key sentence:

> The Agent is not replacing the whole operations system. It is inserted between anomaly detection and human handling, where repeated analysis work is expensive.

## 5. Show Root Cause Evidence

Scroll to:

- `核心结论`
- `关键证据`
- `主要原因排序`
- `责任分发`

Explain:

> GMV drop is not answered as one SQL. The system splits it into order volume, AOV, user scale, category contribution, funnel and refund signals. Recent additions also include business evidence such as product, seller, inventory, fulfillment, marketing and after-sales signals.

## 6. Show Human Control

As `数据分析师`:

1. Click `确认异常`.
2. Optionally click `人工确认发送飞书`.
3. Click `派发责任人`.

Explain:

> Feishu status and process status are separated. Sending a message does not mean the issue is handled. Dispatching changes the workflow and makes the anomaly visible to the responsible business role.

## 7. Switch Business Role

Switch to `平台运营` or another responsible role.

Show:

- the assigned anomaly appears in that role's queue;
- business owner can `接手处理`;
- business owner can add handling notes;
- once the reason is clear, they can close the anomaly.

Key sentence:

> This is why the project is closer to an operations workflow than a normal data Q&A demo.

## 8. Boundaries

Be explicit:

- Olist public data supports order, region, category, product and seller-style drilldown.
- Demo-completed data supports user, funnel, refund and some business evidence.
- Human confirmation is required before notification and dispatch.
- Dedup, read-only protection, timeout and fallback are used to avoid turning the Agent into an incident source.

## 30-Second Summary

> I built an ecommerce operations anomaly analysis Agent on AssistantAgent. The important part is not only natural language Q&A, but the workflow: anomaly ID enters the system, the Agent selects an analysis route, calls tools for evidence, generates root cause and action routing, then humans confirm, dispatch and close the issue. This makes the project demonstrate Agent integration into an existing business system rather than a standalone chatbot.
