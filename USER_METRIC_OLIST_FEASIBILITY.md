# UserMetricTool 迁移到 Olist 的可行性判断

这份文档回答一个很实际的问题：

> 既然 `GMV / 区域 / 品类 / 订单结构` 已经逐步迁到 Olist 支线，为什么 `UserMetricTool` 现在还不适合马上切过去？

## 1. 先说结论

**短期内不建议把 `UserMetricTool` 直接迁到 Olist。**

原因不是“做不到”，而是：

- Olist 更像 **订单经营数据**
- `UserMetricTool` 现在服务的是 **用户规模 / 活跃 / 渠道质量** 这类更偏行为和业务口径的问题

所以如果硬切，最容易出现的问题不是代码报错，而是：

**看起来也能答，但业务口径已经悄悄变了。**

## 2. 当前 `UserMetricTool` 在回答什么

当前主链里，`UserMetricTool` 主要服务两类问题：

1. **用户规模**
   - `昨天 DAU 和活跃买家多少？`
   - `那华东呢？`

2. **渠道质量**
   - `哪个渠道转化更高？`
   - `成交变化是不是先从用户规模或渠道质量开始？`

换句话说，它不是在回答“有多少客户下过单”，而是在回答：

**今天有多少用户在活跃、在购买、不同渠道的用户质量怎么样。**

## 3. Olist 擅长什么

Olist 公开数据很适合承接：

- 区域经营结果
- 订单量
- 支付订单量
- 客单价
- 品类表现
- 商品和订单的经营结构

所以它适合迁：

- `RegionPerformanceQueryTool`
- `OrderQueryTool`
- `CategoryRankTool`

## 4. Olist 为什么不适合直接承接 `UserMetricTool`

### 4.1 没有真正的行为事件流

你现在的 `UserMetricTool` 想回答的是：

- `DAU`
- `活跃买家`
- `渠道转化率`

但 Olist 没有：

- 浏览事件
- 会话事件
- 加购事件
- 用户活跃日志

它更像“订单结果”，不是“用户过程”。

所以它可以近似支持：

- 下单客户数
- 支付客户数

但不天然支持：

- `DAU`
- 真实意义上的“活跃用户”
- 当前主链里的渠道质量口径

### 4.2 当前渠道维度是业务化演示口径

你现在 demo 主链里有：

- `organic`
- `live_stream`
- `ad_feed`

这本质上是在模拟中国电商内部更常见的渠道语义。

而 Olist 公开数据里没有一套同等级的、可直接复用的渠道来源字段。  
如果现在强行切过去，最可能发生的是：

- 为了“能答”临时编出一套渠道口径
- 最后把用户增长分析讲得很虚

### 4.3 root cause 里用户层的任务不是“客户数统计”

在 `gmv_root_cause` 里，用户层的作用是：

**判断问题是不是先从用户规模和渠道质量开始。**

这要求的不只是“有多少人买过”，而是更稳定的：

- 活跃口径
- 用户规模口径
- 渠道质量口径

而这些恰恰是 Olist 当前最弱的一层。

## 5. 如果硬迁，会冒什么业务风险

1. **把“活跃用户”偷换成“下单用户”**
   - 看起来还能答
   - 但指标意义已经变了

2. **把“渠道质量”变成臆造字段**
   - 演示时好像更完整
   - 但经不起追问

3. **让 root cause 用户层结论失真**
   - 最终会把“问题先出在用户规模还是交易结构”讲偏

## 6. 现在最合理的策略

当前更合理的做法是：

- **区域 / 订单结构 / 品类**
  - 先迁到 Olist
- **用户规模 / 漏斗 / 退款**
  - 暂时保留在 demo 主链

这就是为什么现在 `gmv_root_cause` 已经进入：

**混合数据迁移期**

而不是“全链一次性迁移期”。

## 7. 什么时候再考虑迁 `UserMetricTool`

可以等下面其中一类条件成立后再迁：

1. 额外补到更接近行为事件的数据源
   - 比如更像淘宝用户行为的数据

2. 先把 `UserMetricTool` 缩成更保守的订单客户口径
   - 明确不再讲 `DAU`
   - 只讲买家数 / 支付买家数

3. 明确拆成两条线
   - `UserMetricTool` 保留 demo 业务口径
   - 另做一条 `CustomerOrderMetricTool` 走 Olist

## 8. 一句话总结

**Olist 适合迁经营结果分析，不适合立刻迁用户行为口径分析。**

所以当前不迁 `UserMetricTool`，不是因为技术上懒得做，而是因为：

**现在先守住业务口径，比追求“全部公开数据”更重要。**
