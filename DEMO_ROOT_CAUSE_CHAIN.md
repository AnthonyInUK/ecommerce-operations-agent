# GMV Root Cause Demo Chain

这份文档是第一版演示链路说明，目标不是展示“模型会聊天”，而是展示：

**这个 Agent 已经能按固定业务套路做最小归因分析。**

## 1. 演示目标

用一句问题证明系统不只是会报数，而是会沿着业务问题往下拆：

> `2026-05-17 华东 GMV 为什么跌了？`

这条演示链想解决的业务问题是：

- 运营先看到大盘或区域变差
- 不想再等分析师排队查数
- 系统至少先给出一版结构化归因

## 2. 当前演示结论在查什么

当前版本的 `gmv_root_cause` 不再只回答“哪个品类拖后腿”，而是开始按四层解释：

1. **结果层**：华东今天比昨天到底跌了多少
2. **用户规模层**：DAU / 活跃买家有没有先走弱
3. **订单结构层**：订单量 / 客单价有没有走弱
4. **过程和售后层**：漏斗转化有没有掉、退款有没有抬升

所以它已经从“看现象”升级成了：

**用户规模 + 订单结构 + 品类 + 漏斗 + 退款** 的最小归因链。

## 2.1 当前已经进入“混合数据 root cause”阶段

当前这条 root cause 链，不是所有环节都来自同一套数据，而是分阶段迁移：

- **公开数据 Olist**
  - 区域表现
  - 订单结构
  - 品类表现
- **当前 demo 主链**
  - 用户规模
  - 漏斗
  - 退款

业务上可以把它理解成：

> 这条链已经开始吃真实公开订单数据，但还没有为了“全公开数据”牺牲现有用户、漏斗、退款这些更完整的业务口径。

这不是妥协，而是刻意的迁移策略：

1. 先把 **最适合公开订单数据承接** 的部分迁过去
2. 暂时保留 **更依赖业务口径和行为事件** 的部分
3. 用 `data_lineage` 明确告诉观众，每个结论来自哪条数据线

## 3. 当前标准深路径

当前 root cause 走的是一条固定深路径，不是完全开放规划：

1. `RegionPerformanceQueryTool`
2. `OrderQueryTool`
3. `UserMetricTool`
4. `CategoryRankTool`
5. `FunnelAnalysisTool`
6. `RefundAnalysisTool`

## 4. 每一步在业务上排查什么

### Step 1. `RegionPerformanceQueryTool`

先确认：

- 华东今天的 GMV
- 华东前一天的 GMV

业务意义：
先确认异常是真的，不凭感觉立案。

当前数据来源：
- 在 `prefer-olist-analytics=true` 下，优先来自 **Olist 公开数据**

### Step 2. `OrderQueryTool`

再看：

- 订单量是不是掉了
- 客单价是不是掉了

业务意义：
把 GMV 拆成交易结构，回答“是单量问题还是客单价问题”。

当前数据来源：
- 在 `prefer-olist-analytics=true` 下，优先来自 **Olist 公开数据**

### Step 3. `UserMetricTool`

再看：

- DAU 有没有回落
- 活跃买家有没有回落

业务意义：
回答“问题是不是先从用户规模开始”。

当前数据来源：
- **暂时仍来自 demo 主链**

为什么先不切到 Olist：
- Olist 更像订单经营数据，不是真正的用户行为事件流
- 它能支撑“下过单的客户”，但不天然支撑你现在这套 `DAU / 活跃买家 / 渠道转化率` 口径

### Step 4. `CategoryRankTool`

再拆：

- 哪些品类在拖后腿
- 哪些品类相较前一天下降更明显

业务意义：
从区域异常走到业务结构异常。

当前数据来源：
- 在 `prefer-olist-analytics=true` 下，优先来自 **Olist 公开数据**

### Step 5. `FunnelAnalysisTool`

再排查：

- 浏览到支付的转化有没有掉

业务意义：
区分这是流量问题还是转化问题。

当前数据来源：
- **暂时仍来自 demo 主链**

为什么先不切到 Olist：
- Olist 没有完整的浏览、加购、下单、支付前链路事件
- 所以不适合直接承接现在这套 `view->pay / order->pay` 漏斗口径

### Step 6. `RefundAnalysisTool`

最后看：

- 售后/退款是不是把结果进一步拖差

业务意义：
如果前面都还行，最后要排除是不是成交后质量问题。

当前数据来源：
- **暂时仍来自 demo 主链**

为什么先不切到 Olist：
- Olist 没有干净直接的退款事实层
- 当前 demo 里的退款语义链更完整，更适合先承接 root cause 最后一段

## 5. 当前回答结构

当前系统返回的 root cause 回答，不是原始数据堆砌，而是按固定结构组织：

1. 异常确认
2. 用户规模判断
3. 订单结构判断
4. 主要拖累品类
5. 漏斗判断
6. 退款判断

并且现在会显式带出一层 `data_lineage`：

- `region_metrics_source`
- `order_metrics_source`
- `user_metrics_source`
- `category_metrics_source`
- `funnel_metrics_source`
- `refund_metrics_source`

业务意义：
不是只告诉你“结论是什么”，还告诉你“这条结论目前来自公开数据还是 demo 业务口径”。

也就是：

> 华东在 2026-05-17 相比 2026-05-16 的 GMV 从 X 下降到 Y。  
> 用户规模侧……；交易结构侧……；按品类拆解……；漏斗侧……；售后侧……

## 6. 演示时怎么讲

### 开场一句话

> 这不是一个“会报数”的 Agent，我拿一个最典型的问题演示它怎么做最小归因：`华东 GMV 为什么跌了？`

### 如果面试官追问“公开数据接到哪一步了”

可以直接这样讲：

> 这条 root cause 现在已经进入混合数据迁移期。区域、订单结构、品类三段优先切到 Olist 公开数据，因为它很适合承接订单经营分析；用户规模、漏斗、退款还先留在 demo 主链，因为这些更依赖行为事件和业务口径。  
> 我没有为了“全部公开数据”牺牲当前分析质量，而是先迁最适合迁的那三段，并且用 `data_lineage` 和启动校验把这条迁移线守住。

### 口播顺序

1. 先问一句：`昨天 GMV 多少？`
2. 再问一句：`昨天订单量多少，客单价怎么样？`
3. 再问一句：`昨天 DAU 和活跃买家多少？`
4. 最后问：`2026-05-17 华东 GMV 为什么跌了？`

这样观众能先看到：

- 大盘
- 交易结构
- 用户规模
- 最终归因

### 你要强调的不是“模型懂了”

而是：

> 系统已经把这类高频复杂问题沉淀成了一条固定分析套路，后面会继续交给 Experience / FastIntent / verified cases / evaluation summary 去复用和评估。

## 7. 这条演示链证明了什么

它证明的不是“Agent 已经完全通用”，而是：

1. 这个项目已经不是单纯 NL2SQL 或看板问答
2. 它开始会做受控的多步业务分析
3. 它已经能把高频复杂问题固化成标准深路径

这正是后面继续做：

- `verified cases`
- `FastIntent`
- `evaluation summary`
- `runtime metrics`

的基础。
