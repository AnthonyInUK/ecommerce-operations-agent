### P1 NL2SQL 准确率天花板

#### 数据来源（可信，引用时注明）

| 来源 | 关键数据 |
|------|---------|
| **Spider 2.0 基准（2024，UC Berkeley/Stanford/Georgia Tech）** | 600 道企业级 SQL 题，GPT-4 仅答对 **6%** |
| **BIRD 基准** | 最优模型 65.45 分 vs 人类 92.96 分，**差距 27 分** |
| **TAG 论文（2024）** | SQL 无法表达语义推理，需要外部知识的查询正确率 **< 20%** |
| **IDC 2024 中国 BI 调研** | 70% 企业反映 AI 查询结果不准确；60% 用户因 SQL 门槛放弃 BI |
| **Snowflake 官方工程 Blog（2025）** | 有语义层可达 95%，无上下文裸 SQL 仅 50–70%，**AtScale 测试无语境下仅 16%** |

#### 现象（Silent Failure 最致命）

SQL 生成结果不报错但业务上是错的。例如：
- Snowflake Cortex Analyst 文档明确提到："询问'预测准确率'，模型用百分比差值计算，但业务上定义的是绝对误差"——**技术上正确，业务上错误**。
- Databricks Genie 故障排查文档：WHERE 条件过滤值不匹配（数据库存的是"CA"，模型写的是"California"）。

#### 各产品现状

| 产品 | 方案 | 缺陷 |
|------|------|------|
| **Coze（字节）** | NL2SQL 插件 | 依赖预定义数据库 Schema，复杂 join 失败率高；无法做多步骤计算 |
| **Dify** | Text-to-SQL 工作流 | 准确率依赖提示词调优，用户需手写大量 Schema 描述；错误无法自校验 |
| **FineBI（帆软）** | 自然语言转 BI 图表 | 仅支持其自有数据集，语义解析局限；SegmentFault 横评指出歧义问题明显 |
| **Quick BI（阿里）** | NL2SQL | 企业级 Schema 复杂时失败率高；需要维护庞大的数据字典 |
| **瓴羊智能小Q（阿里）** | 自研语义层 + SQL | InfoQ 专访：企业行业 know-how 积累是核心挑战，精准度是企业级难题 |
| **Databricks Genie（美）** | Text-to-SQL + 验证层 | Token 限制导致大表查询截断；WHERE 值匹配依赖数据可见性 |
| **Snowflake Cortex（美）** | Text-to-SQL + 语义模型文件（YAML）| 必须维护语义层才能达到可用精度，冷启动（无语义层）仅 16–54% |
| **Microsoft Fabric Copilot（美）** | GPT-4 + Power BI 数据模型 | 非确定性输出，官方文档标注"不提供准确性保证"；不能计算模型外字段 |

#### 我的项目如何解决

**Text2Code 取代 Text2SQL**：LLM 生成 Python 代码，在 GraalVM 沙箱执行，中间可以：
- 分步查询，对结果做二次验证
- 发现数据格式不匹配时自动修正（如 "CA" vs "California"）
- 用代码逻辑表达 SQL 无法表达的语义推理（窗口函数、条件分支、业务规则）

### P2 单步问答，无法多步推理

#### 数据来源

| 来源 | 关键数据 |
|------|---------|
| **TAG 论文（2024）** | 传统 RAG+SQL 对多跳推理问题正确率 < 20% |
| **Gartner 2025** | 超过 40% 的 Agentic AI 项目将在 2027 年前被取消，原因之一是"单步工具调用无法完成复杂分析任务" |
| **Statology Agentic Analytics 2025 报告** | 8/10 企业反映 AI 分析系统无法完成跨多个数据源的推理链 |
| **53AI / 知乎 Coze 用户反馈** | "涉及复杂任务时稳定性差，多步骤工作流经常在中途失败" |

#### 典型失败场景

```
用户问：上个月哪些门店的 GMV 环比下降超过 20%，下降原因是什么？
NL2SQL 路径：生成一条 SQL → 得到 GMV 数字 → 结束（无法分析原因）
Text2Code 路径：
  Step 1. 查上月和上上月门店 GMV
  Step 2. 计算环比，筛选 > 20% 的门店
  Step 3. 拉这些门店的退款率、客单价、活动情况
  Step 4. 用代码对比异常指标，生成归因报告
```

#### 各产品现状

| 产品 | 限制 |
|------|------|
| **Coze** | 工作流节点拼接可以多步，但每步之间无状态传递；复杂任务节点增多后失败率高 |
| **Dify** | 工作流支持多步，但分支条件需手写，无法自动规划步骤 |
| **Databricks Genie** | 单次 Text-to-SQL，不支持 Agent 自主规划多步骤查询 |
| **Snowflake Cortex** | 每次请求独立无状态；"无法访问过去的查询结果" |
| **Microsoft Fabric Copilot** | 官方文档："Copilot 不能理解之前的输入，也不能撤销已提交的更改" |

#### 我的项目如何解决

AssistantAgent **Code-as-Action** 天然支持多步推理：LLM 生成完整 Python 脚本，脚本内可以写 for loop、if/else、多次 API 调用，GraalVM 统一执行，结果自动传递给下一步。

### P3 无会话记忆，无法多轮对话

#### 数据来源

| 来源 | 关键数据 |
|------|---------|
| **Snowflake Cortex 官方文档** | "Cortex Analyst 是无状态的，每次请求所有上下文必须显式带入" |
| **Microsoft Fabric Copilot 官方文档** | "Copilot 不能理解之前的输入" |
| **瓴羊白皮书（2025）** | 明确提出"可行动"作为最高演进阶段，当前产品仍卡在"可决策"阶段 |
| **V2EX Dify 讨论帖（2024）** | 用户抱怨"每次问问题都要重新说背景，没有上下文记忆" |

#### 典型场景

```
用户：上月北京的 GMV 是多少？
AI：3200 万。
用户：那上海呢？   ← AI 不知道"那"指代什么，返回错误或重复询问
用户：比上上月涨了多少？ ← AI 已忘记前两个问题的数字
```

#### 各产品现状

| 产品 | 记忆能力 |
|------|---------|
| **Coze** | Session 内有消息历史，但对大量历史上下文处理效果差 |
| **Dify** | 依赖外挂 Memory 组件，配置复杂，非默认开启 |
| **Databricks Genie** | 支持对话历史，但跨 Genie Space 无记忆 |
| **Snowflake Cortex** | **完全无状态**，官方明确声明 |
| **FineBI** | 报表式交互，无对话设计 |

#### 我的项目如何解决

AssistantAgent **Session 级代码上下文（context 模块）**：每次执行的中间变量、查询结果可持久化到 Session 上下文，下一轮提问时自动注入，实现真正的多轮分析对话。

### P4 指标歧义与语义理解偏差

#### 数据来源

| 来源 | 关键数据 |
|------|---------|
| **AtScale × Snowflake 测试（2025）** | "预测准确率"的不同业务定义导致 SQL 正确但业务答案错误 |
| **IDC 2024** | 45% 企业用户反映 AI 无法理解口语化提问（"销量还行吗" 等） |
| **瓴羊 InfoQ 专访（2025）** | "企业行业 know-how 积累是最大挑战"；同一词汇在不同部门意思不同 |
| **SegmentFault BI 横评（2026）** | FineBI 已引入"歧义澄清机制"，说明这是行业公认痛点 |

#### 典型歧义场景

- "销售额" = 含税？不含税？
- "活跃用户" = DAU？近 30 天登录过？购买过的？
- "增长率" = 环比？同比？CAGR？
- "上月" = 自然月？最近 30 天？上个财务月？

#### 各产品现状

| 产品 | 歧义处理 |
|------|---------|
| **Coze** | 无；模型自行猜测，结果不可预知 |
| **Dify** | 依赖 Prompt 工程，需手动写"指标定义"文档 |
| **FineBI** | 引入了澄清问答机制（行业率先），但仅限自有数据集 |
| **Snowflake Cortex** | 依赖语义模型 YAML 文件预定义指标，冷启动完全不处理歧义 |
| **Quick BI** | 需维护庞大指标字典；更新滞后于业务变化 |

#### 我的项目如何解决

**两层策略：**
1. **经验体系预置指标定义**：常用指标的计算逻辑存入 Experience（COMMON 类型），LLM 调用时自动注入
2. **多轮澄清**：检测到歧义时生成追问，用户确认后执行（模仿 FineBI，但扩展到开放数据源）

### P5 分析过程黑盒，无法溯源

#### 数据来源

| 来源 | 关键数据 |
|------|---------|
| **Deloitte 2024 调研** | 38% 高管反映曾基于 AI 分析做出错误决策 |
| **McKinsey 2025** | 仅 1% 企业认为 AI 已达到可信任程度；可解释性是核心障碍 |
| **Microsoft Fabric Copilot 官方文档** | "不建议在自主、高风险或业务关键流程中使用 Copilot" |
| **瓴羊白皮书（2025）** | 企业痛点之一："到底有没有数"—— AI 给了个数，但我不知道它从哪来 |

#### 典型场景

```
AI 回答：上月北京 GMV 3,200 万
用户无法知道：
  - 查了哪张表？
  - WHERE 条件是什么？
  - 是否排除了退款订单？
  - 结果是否被四舍五入或截断？
```

#### 各产品现状

| 产品 | 溯源能力 |
|------|---------|
| **Coze** | 无；只返回最终答案 |
| **Dify** | 可查看节点日志，但非标准化，用户需自己理解 |
| **Databricks Genie** | 新版显示"思考步骤"和使用的表、示例 SQL（2024 更新） |
| **Tableau Pulse** | 只推送洞察结论，无法追溯底层数据 |
| **FineBI** | 图表可下钻，但 AI 对话部分无溯源 |

#### 我的项目如何解决

**代码即溯源**：GraalVM 执行的 Python 代码本身就是完整的分析过程，可以：
- 把生成的代码直接展示给用户（分析过程透明）
- LoggerBridge 记录每一步执行日志
- OpenTelemetry 可观测性（observation 模块）自动追踪

### P6 从洞察到行动断链

#### 数据来源

| 来源 | 关键数据 |
|------|---------|
| **瓴羊 InfoQ 专访（VP 级，2025）** | "从洞察到行动是当前行业最前沿、最未解决的课题" |
| **瓴羊白皮书三阶段演进** | 可视化 → 可决策 → **可行动**（绝大多数产品止步于第二阶段） |
| **Gartner 2025 Agentic Analytics 报告** | Agentic Analytics 的核心价值在于"自动触发业务动作"，但落地率极低 |
| **Statology 2025 真实落地报告** | "专用领域的行动触发系统提供了实质价值，但通用 BI 工具几乎无一能做到" |

#### 典型场景

```
现状：每天早上分析师打开 BI 看昨日数据 → 发现异常 → 写邮件通知业务 → 业务讨论 → 决策
理想：Agent 发现 GMV 异常 → 自动生成归因报告 → 推送钉钉/飞书 → 带可操作建议
      或：库存跌破阈值 → Agent 自动发起补货申请流程
```

#### 各产品现状

| 产品 | 行动能力 |
|------|---------|
| **Coze** | 可以发消息通知，但分析→通知是手动工作流，不能自主判断触发 |
| **Dify** | 无内置触发器；需接外部 Webhook |
| **Tableau Pulse** | 推送 AI 洞察摘要（邮件/Slack），但仅推送，不触发业务动作 |
| **Databricks Genie** | 纯问答产品，无主动触发能力 |
| **Quick BI / FineBI** | 数据预警功能，但规则固定，无 AI 推理触发 |

#### 我的项目如何解决

AssistantAgent **Trigger 模块**（定时/延迟/事件触发）+ **Reply 模块**（钉钉/飞书/微信工作群）原生支持：
1. 定时触发（每天凌晨跑分析）
2. 条件触发（GMV 环比下跌 > X% 时触发）
3. 多渠道推送（钉钉/飞书/邮件）
4. 推送内容包含 AI 生成的归因分析 + 可操作建议

### P7 重复问题每次从零推理

#### 数据来源

| 来源 | 关键数据 |
|------|---------|
| **通用观察（多来源）** | 企业中相同分析任务反复执行（日报/周报/月报），每次调用 LLM 全量推理，成本高且延迟大 |
| **瓴羊白皮书** | 企业知识沉淀是长期价值，但现有产品无法积累 |

#### 典型场景

```
每周一问："各城市上周 GMV TOP10 是哪些？"
每月问："上月各品类销售额及环比增长"
→ 每次都要 LLM 推理 → 慢（3-8 秒）+ 贵（token 费用）+ 不稳定（可能得到不同结果）
```

#### 各产品现状

| 产品 | 复用能力 |
|------|---------|
| **Coze** | 无；每次全量推理 |
| **Dify** | 无；每次全量推理 |
| **所有 NL2SQL 产品** | 无经验复用机制 |

#### 我的项目如何解决

AssistantAgent **FastIntent + Experience 体系**：
- 语义相似的问题命中已有 Experience → 直接填充参数执行，跳过 LLM 推理
- **FastIntent ≠ 缓存结果**：提取新参数（如"上周"→新日期范围），执行得到实时数据
- 经验分三类：COMMON（通用规则）、REACT（工具调用模板）、TOOL（代码模板）

### P8 中文口语理解差 / 本土化不足

#### 数据来源

| 来源 | 关键数据 |
|------|---------|
| **IDC 2024 调研** | 45% 用户因 AI 无法理解口语化提问放弃使用 |
| **V2EX Dify 帖（2024，12.3k 浏览）** | 中国用户抱怨网络问题（访问海外 API）、本土模型支持差 |
| **通用观察** | "上月"/"上上月"/"大前天"/"Q3"等时间表达，中英文模型理解差异显著 |

#### 典型场景

- "昨天北京的单量咋样" → AI 无法解析"单量"="订单量"
- "销量还行吗" → AI 无法定义"还行"的判断标准
- "上上个月" → 部分 NL2SQL 引擎不能正确推算日期

#### 各产品现状

| 产品 | 中文支持 |
|------|---------|
| **Coze（字节）** | 字节自研模型，中文理解相对好，但业务词汇仍需配置 |
| **瓴羊智能小Q** | 阿里系，中文支持强，内置行业词库 |
| **Dify** | 开源，模型切换灵活，但默认英文友好 |
| **Databricks Genie** | 英文优先设计，中文支持有限 |
| **Snowflake Cortex** | 英文优先，中文企业需额外配置 |

#### 我的项目如何解决

- 接国产大模型（通义/文心等）作为可选后端
- Experience 体系存储中文业务术语映射（"单量" = 订单数，"客单价" = GMV/订单数）
- FastIntent 的语义匹配对中文 embedding 友好（支持 sentence-transformers 中文模型）

### P9 数据孤岛跨源分析困难

#### 数据来源

| 来源 | 关键数据 |
|------|---------|
| **Statology Agentic Analytics 2025** | 8/10 企业面临数据孤岛问题；AI 无法跨多个系统推理 |
| **McKinsey 2025** | 碎片化数据是 Agentic AI 落地最大障碍 |
| **瓴羊白皮书（2025）** | "泛滥成灾"：数据多但分散，AI 无法整合 |

#### 各产品现状

| 产品 | 跨源能力 |
|------|---------|
| **Coze** | 多插件可接入不同数据源，但跨源 join 推理困难 |
| **Dify** | 工作流可配置多节点，但需手工设计跨源路由 |
| **FineBI / Quick BI** | 仅支持已接入的数据集，跨库 join 需 ETL 预处理 |
| **Databricks Genie** | 基于 Unity Catalog，跨表联查能力强，但仍限于 Databricks 生态 |

#### 我的项目如何解决

Python 代码可以同时连接多个数据源（MySQL + API + CSV + Redis），在代码层做 join 和聚合，不需要提前 ETL。GraalVM 沙箱内的 Python 环境可以 pip 安装任意 connector。

### P10 平台部署 / 私有化困难

#### 数据来源

| 来源 | 关键数据 |
|------|---------|
| **V2EX Dify 踩坑帖（2024，12.3k 浏览）** | 部署地狱：Docker 版本不兼容、依赖缺失、环境变量设置混乱 |
| **V2EX Dify 帖续集** | 插件安装失败、文档与实际行为不符、breaking change 无迁移指南 |
| **通用观察** | 中大厂优先私有化部署，SaaS 产品（Coze）无法满足数据合规要求 |

#### 各产品现状

| 产品 | 私有化支持 |
|------|-----------|
| **Coze** | 纯 SaaS，数据不出字节，中大厂无法接受 |
| **Dify** | 开源可私有化，但部署体验差，V2EX 大量吐槽 |
| **瓴羊智能小Q** | 企业版可私有化，但重，需专业交付 |
| **FineBI / Quick BI** | 成熟私有化方案，但传统 BI 形态 |
| **Databricks / Snowflake** | 云服务，中国数据出境合规问题 |

#### 我的项目如何解决

AssistantAgent 基于 **Spring Boot**（企业友好）+ **GraalVM**（无外部运行时依赖）：
- Spring Boot Starter 自动配置，`mvn clean package` 一键构建
- Docker 镜像自包含，私有化部署简单
- 不依赖外部 SaaS，数据完全在用户环境内

## 三、国外产品痛点专项

> 补充国际视角，验证痛点的全球普遍性（不是中国特色问题）。

### 3.1 Databricks AI/BI Genie

**定位：** Databricks 的对话式分析产品（2024 GA），Text-to-SQL 方向。

| 痛点 | 具体表现 | 来源 |
|------|---------|------|
| **Token 上下文限制** | 大表查询时列描述超出 token 窗口，Genie 开始截断上下文，精度下降 | Databricks 官方 Release Notes 2024 |
| **WHERE 值盲区** | 无法看到数据实际值时，模型猜测过滤条件（用 "California" 而不是 "CA"）| Databricks Troubleshooting 文档 |
| **无多步骤规划** | 每次问答一条 SQL，无法自主规划"先查A，根据A的结果再查B"的分析链 | 产品设计观察 |
| **不懂业务语义** | 纯 SQL 生成，无业务指标层；同一词汇不同团队定义不同时无法区分 | Databricks 社区文章 |
| **边界：仅 Databricks 生态** | 只能查 Unity Catalog 内的数据，无法跨源 | 产品文档 |

**官方承认的改进方向（2025 Genie Next Gen）：** 增加"Thinking Steps"展示（溯源改进）、智能列过滤（解决 token 截断）。

---

### 3.2 Snowflake Cortex Analyst

**定位：** Snowflake 的自然语言分析接口，依赖语义模型文件（YAML）。

| 痛点 | 具体表现 | 来源 |
|------|---------|------|
| **完全无状态** | 每次对话独立，无法记住上一问题的结果；多轮分析需每次把全部上下文手动带入 | Snowflake 官方文档 |
| **冷启动精度极低** | 无语义层裸用：准确率 16%–54%；加语义层后 86–95%。问题：语义层维护成本极高 | AtScale 测试报告（2025） |
| **语义层维护成本** | 需要专业数据工程师维护 YAML 语义模型，随业务变化需持续更新 | AtScale Blog |
| **业务定义不统一** | "预测准确率"不同部门有不同定义，Cortex 无法自动识别应用哪个定义 | AtScale Blog |
| **Ground Truth 时效性** | 时间相关的验证查询（"上月"）随时间推移结果漂移 | Snowflake 官方文档 |
| **模型方言适配** | 各 LLM 对 Snowflake SQL 方言理解参差不齐，需大量提示词工程 | Snowflake 工程 Blog |

---

### 3.3 Microsoft Fabric Copilot

**定位：** Microsoft 一站式数据分析平台内嵌的 AI 助手。

| 痛点 | 具体表现 | 来源 |
|------|---------|------|
| **非确定性输出** | LLM 非确定性，官方明确声明"不提供准确性、可靠性或真实性保证" | Microsoft 官方文档 |
| **无法跨数据模型边界** | 只能操作已在 Power BI 模型中定义的 Measures 和 Fields；无法计算模型外字段 | Fabric 社区讨论 |
| **完全无状态** | 不理解之前的输入；无法撤销已提交的更改 | Microsoft 官方文档 |
| **单请求限制** | 不能批量操作（如"把所有查询的列标题大写"需逐个处理） | Fabric 社区 |
| **算力成本高** | Output token 价格是 Input token 的 3 倍；复杂分析 token 消耗大 | Microsoft 文档 |
| **不适合高风险场景** | 官方声明"不建议用于自主、高风险或业务关键流程" | Microsoft 文档 |

---

### 3.4 Tableau Pulse / Tableau AI

**定位：** Salesforce/Tableau 的 AI 驱动主动推送型分析产品。

| 痛点 | 具体表现 | 来源 |
|------|---------|------|
| **推送型而非对话型** | 只能订阅指标、接收推送的 AI 摘要；无法主动提问追问 | 产品设计观察 |
| **自定义分析能力弱** | 洞察由 Einstein AI 自动生成，用户无法自定义分析逻辑 | Tableau 文档 |
| **无行动触发** | 只推送摘要，不能触发下游业务系统（如自动发起工单、补货） | 产品设计观察 |
| **日历对齐问题** | 非标准日历（如财务月、数据非每日更新）时"今天"的定义会出错（已在 2025 更新修复） | Tableau Release Notes 2025 |
| **Salesforce 生态锁定** | 深度依赖 Salesforce + Tableau 数据生态，跨系统集成复杂 | 产品设计观察 |

### 3.5 通用 Agentic Analytics 行业痛点（来源：Gartner、McKinsey、Statology）
