# AssistantAgent 电商分析 Agent 系列 04：为什么业务型 Agent 的 Tool 不该按表切

## 背景

前面几篇里，我已经把这个电商分析 Agent 的主线讲到三个层次：先从问答 Demo 做到最小业务闭环，再解决多轮追问里的最小业务记忆，接着把“为什么跌了”收成一条标准归因链。

这篇回头补一个更底层、但很关键的问题：Tool 到底应该怎么设计。

做数据分析 Agent 时，很容易把 Tool 按数据库表来切。比如电商里有订单表、商品表、支付表、退款表、用户表，于是很自然地会想到：

```text
OrdersTool
ProductsTool
PaymentsTool
RefundsTool
UsersTool
```

这种切法对开发者很直观，因为它贴近数据表结构。但对业务型 Agent 来说，它不一定是最好的抽象。

业务人员问的通常不是“帮我查 orders 表”，而是：

```text
昨天 GMV 多少？
订单量和客单价怎么样？
哪个区域拖后腿？
哪个品类掉得最多？
退款是不是变严重了？
为什么跌了？
```

这些问题天然不是表名，而是分析动作。也正因为这样，我在这个项目里没有把第一批 Tool 按表切，而是按业务分析动作切。

## 按表切的问题在哪里

按表切最大的问题，是它把复杂度留给了模型。

假设系统只有 `OrdersTool`、`ProductsTool`、`PaymentsTool`、`RefundsTool`，用户问“华东 GMV 为什么跌了”，模型就必须自己决定：

- 先查哪个表
- 用什么字段算 GMV
- 按下单时间还是支付时间
- 订单状态怎么过滤
- 退款要不要扣减
- 区域字段从哪里来
- 品类字段怎么关联
- 最后怎么把多张表的结果组织成原因

这不是不能做，但会让模型承担太多业务口径和数据建模细节。短期看起来灵活，长期很容易出现三个问题。

第一个问题是口径漂移。同一个 GMV，今天可能按支付金额算，明天可能把退款扣掉，后天可能误用下单时间。只要这些口径没有被 Tool 固化，模型每次生成代码时都有机会漂。

第二个问题是路径不稳定。用户问“订单量怎么样”，模型可能只查订单表；用户问“GMV 为什么跌了”，模型又要自己决定订单、用户、品类、退款之间的顺序。这样后面很难做标准快路径和标准深路径。

第三个问题是评测困难。如果 Tool 粒度太贴近底层表，评测时很难判断一次回答到底有没有完成业务动作。系统可能查了表，但没有完成“判断是订单量问题还是客单价问题”这个分析任务。

所以按表切不是错，而是更适合作为数据访问层。对业务型 Agent 暴露出来的 Tool，最好不要停留在表层。

## 按业务动作切，模型才更容易稳定

当前项目里的第一批 Tool 是这样设计的：

| Tool | 解决的业务问题 |
| --- | --- |
| `GmvQueryTool` | 昨天卖了多少，核心盘子怎么样 |
| `OrderQueryTool` | GMV 变化到底是订单量变了，还是客单价变了 |
| `UserMetricTool` | 成交变化是不是先从用户规模或渠道质量开始 |
| `RegionPerformanceQueryTool` | 哪个区域更差，哪个大区需要先复盘 |
| `CategoryRankTool` | 哪些品类在拉动或拖累大盘 |
| `FunnelAnalysisTool` | 问题是流量少了，还是转化掉了 |
| `RefundAnalysisTool` | 成交后是不是售后或退款在拖后腿 |

这组 Tool 的命名并不完全贴近数据库表，而是贴近业务问题。

比如 `OrderQueryTool` 不是“查订单表”，而是直接服务一个分析动作：判断 GMV 变化到底是订单量问题，还是客单价问题。它的输出也不只是订单明细，而是订单量、支付订单量、退款订单量、支付金额和客单价。

`CategoryRankTool` 也不是“查商品表”，而是服务品类结构分析。用户问品类排行、品类拖累、区域内哪个品类更差，都可以从这个动作开始。

`RefundAnalysisTool` 也不是“查退款记录”，而是服务售后压力判断。它返回的是退款金额、关联 GMV、退款金额占比，以及风险集中在哪些品类。

这带来一个很直接的好处：模型调用 Tool 时，不需要重新理解底层表结构，而是在调用一个已经带有业务语义的分析动作。

## Tool 描述本身就是 Prompt 的一部分

在 Agent 系统里，Tool 不是只有代码实现有价值。Tool 的名字、描述、参数、返回值，其实都会影响模型怎么调用它。

项目里的 Tool 都继承了一个公共基类 `AbstractWarehouseQueryCodeactTool`，每个 Tool 都会提供几类信息：

```text
tool name
description
input schema
CodeactToolDefinition
CodeactToolMetadata
few-shot example
```

这意味着 Tool 不是简单的 Java 方法，而是一个带语义说明的能力单元。

比如 `OrderQueryTool` 的描述不是“查询订单数据”，而是：

```text
查询某一天的订单结构指标，可按区域或品类筛选，返回订单量、支付订单量、退款订单量、总支付金额和客单价，适合拆解 GMV 变化到底是订单量还是客单价驱动。
```

这段描述很长，但它很重要。因为模型看到这段描述后，会更容易知道什么时候应该调用它。用户问“订单量和客单价怎么样”，它可以命中；用户问“GMV 为什么跌了”，在 root cause 链里也应该调用它。

所以 Tool 设计不是简单写一个函数，而是在给模型定义一组可理解、可组合、可约束的业务动作。

## 快路径和深路径都依赖 Tool 粒度

按业务动作切 Tool 的另一个好处，是它天然支撑快路径和深路径。

快路径解决的是高频标准问题：

```text
昨天 GMV 多少？              -> GmvQueryTool
昨天订单量多少，客单价怎么样？ -> OrderQueryTool
昨天 DAU 和活跃买家多少？     -> UserMetricTool
华东和华南哪个更差？          -> RegionPerformanceQueryTool
品类排行怎么看？              -> CategoryRankTool
```

这些问题不应该每次都让模型从零规划。因为它们很常见，业务口径也比较稳定。只要 Tool 粒度设计得足够贴近问题，FastIntent 或标准路由就可以直接把问题分发到对应 Tool。

深路径解决的是复杂归因问题，比如：

```text
华东 GMV 为什么跌了？
```

这时系统可以把多个业务动作串起来：

```text
RegionPerformanceQueryTool
-> OrderQueryTool
-> UserMetricTool
-> CategoryRankTool
-> FunnelAnalysisTool
-> RefundAnalysisTool
```

如果这些 Tool 一开始是按表切的，root cause 就会变成“模型自己写一堆跨表查询”。但现在每个 Tool 都对应一个分析动作，深路径就更像把多个分析动作组合起来，而不是临时拼 SQL。

这也是我后面能把 `gmv_root_cause` 固化成标准归因链的基础。

## Tool 背后仍然要有稳定数据层

按业务动作切 Tool，并不意味着数据层可以随便写。

当前项目里，这些 Tool 不是直接把原始表暴露给模型，而是统一走 `JdbcWarehouseQueryService`。也就是说，模型看到的是 `GmvQueryTool`、`OrderQueryTool`、`CategoryRankTool` 这些业务动作；Tool 背后再去读 demo 数仓或 Olist 加工后的 ads / dwd 层。

这层隔离很重要。

第一，它避免模型直接面对原始表。原始表字段多、口径散，直接暴露给模型会放大幻觉和误用风险。

第二，它让数据迁移更可控。后来我接 Olist 公开数据时，并没有推翻 Agent 主链，而是先把区域、订单、品类这些更适合公开数据承接的部分迁到 Olist 支线。因为 Tool 的业务接口没变，底层数据来源可以逐步替换。

第三，它让页面、日报、异常巡检能复用同一组业务能力。只要 Tool 输出结构稳定，上层是手动问答、trigger 还是通知草稿，都可以复用。

所以这里真正的分层是：

```text
用户问题
-> 业务动作 Tool
-> 稳定查询服务
-> dwd / dim / ads 数据层
-> raw 原始数据
```

模型不应该直接从用户问题跳到 raw 表。中间这层业务动作 Tool，就是让 Agent 稳定起来的关键。

## Tool 返回值也要为后续链路服务

一开始很容易只关注 Tool 能不能查到数，但后面会发现，Tool 返回什么结构同样重要。

如果 Tool 只返回一段自然语言，后续就很难继续组合。比如 `OrderQueryTool` 如果只返回“订单量下降了”，那 root cause 后面就很难继续比较订单量、客单价、退款订单量这些细项。

所以当前 Tool 基本都返回结构化结果，包括：

```text
success
analysis_space
query_type
stat_date
region_name / category_l1
data_source
rows
count
```

这些字段看起来很工程化，但它们让后面的链路更容易继续处理。

`analysis_space` 可以告诉系统这是大盘、增长、退款还是 root cause。  
`query_type` 可以让页面和评测知道这次查的是哪类问题。  
`data_source` 可以承接 Olist + demo 混合数据口径。  
`rows` 保留结构化事实，方便后续 builder 继续生成原因排序、证据卡片和责任分发。

也就是说，Tool 不只是给用户看的，它还要给后面的 Agent 链路看。

## 这里踩过的坑

第一个坑，是一开始容易把 Tool 做得太像 DAO。比如一个 Tool 只负责查某张表，返回一堆字段。这样开发很快，但模型拿到的只是数据能力，不是分析能力。

第二个坑，是 Tool 描述写得太短。很多时候不是模型“不聪明”，而是 Tool 描述没有告诉它这个能力该在什么业务场景下使用。像“查询订单数据”就不如“判断 GMV 变化是订单量还是客单价驱动”有效。

第三个坑，是 Tool 输出如果不结构化，后面很难复用。最开始看起来只要能回答问题就行，但做到 root cause、页面展示、异常巡检和通知草稿后，就会发现结构化输出比一段漂亮文案更重要。

第四个坑，是不要把所有东西都做成一个万能 Tool。比如可以做一个 `EcommerceAnalysisTool` 接收任意问题，但那样会把路由、口径、评测和复用都塞进一个黑盒里。拆成多个业务动作 Tool，反而更容易逐步验证。

## 当前边界

这套 Tool 设计还不是完整的电商分析平台能力。

当前覆盖比较稳定的是 GMV、订单结构、用户规模、区域、品类、漏斗和退款。广告 ROI、履约物流、商家治理、库存、活动复盘这些还没有完整展开。

另外，按业务动作切 Tool 也不是说永远不需要表级能力。更准确地说，表级能力应该留在数据访问层或查询服务里；Agent 层暴露给模型的，优先应该是业务动作。

这个边界很重要。因为业务动作 Tool 不是为了隐藏所有复杂性，而是为了把最容易出错的指标口径、维度筛选和分析语义先固定住。

## 总结

做完这批 Tool 后，我对业务型 Agent 的一个判断是：Tool 设计决定了系统后面能走多远。

如果 Tool 按表切，系统看起来很灵活，但模型需要承担太多数据口径和分析路径选择。  
如果 Tool 按业务动作切，系统一开始看起来没那么“万能”，但更容易沉淀快路径、标准深路径、评测样例和数据迁移策略。

对电商分析 Agent 来说，真正有价值的 Tool 不是 `OrdersTool`、`ProductsTool` 这种表级能力，而是 `OrderQueryTool`、`CategoryRankTool`、`RefundAnalysisTool` 这种能直接承接业务问题的分析动作。

这也是这篇文章想表达的核心：业务型 Agent 的 Tool 不是数据库接口，而是可复用的业务分析动作。只有这层设计稳住了，后面的多轮追问、root cause、Experience、FastIntent、异常巡检和通知推送才有稳定的基础。
